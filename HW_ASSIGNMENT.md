# Домашнє завдання: Production-Ready AgentCore Deployment

## Контекст
Це домашнє завдання є прямим продовженням воркшопу й має бути побудоване на тій самій архітектурі, яку ви вже реалізували:
- AgentCore Runtime
- Inbound Auth (Cognito JWT)
- Gateway
- Outbound OAuth до Google Docs
- Observability

Не змінюйте архітектуру на інший стек. Мета — довести воркшопне рішення до production-ready рівня.

## Обов'язково перед стартом
Ознайомтесь з інструкцією з MCP-інструментів:
- [Агентський кодінг (MCP setup)](docs/агентський%20кодінг.md)

## Мета
Перетворити E2E-воркшоп рішення на відтворюваний, безпечний, спостережуваний і надійний деплоймент.

## Рівні завдання
### Рівень 1 (обов'язковий)
Написати test cases для E2E-флоу на базі воркшопної архітектури:
- AgentCore Runtime
- Inbound Auth (Cognito JWT)
- Gateway
- Outbound OAuth до Google Docs

Мінімальний набір test cases має покривати:
- авторизацію (успішний і negative auth сценарій);
- runtime-виклик із tool-calling;
- consent/path поведінку (або коректну обробку вже виданого consent);
- базовий failure/fallback сценарій.

### Рівень 2 (обов'язковий, після Рівня 1)
Підготувати документ, який пояснює:
- як тестувати рішення крок за кроком;
- як використовувати різні датасети для тестування;
- які очікувані результати для кожного типу датасету;
- як інтерпретувати помилки і відхилення в результатах.

### Рівень 3 (обов'язковий, після Рівнів 1-2)
Розгорнути додаток і виконати тести в реальному середовищі:
- runtime-агент задеплоєний через `BedrockAgentCoreApp`, `@app.entrypoint` та `app.run()`;
- Gateway і Google Docs target налаштовані та працездатні;
- є докази виконання тестів після деплою;
- є observability-докази щонайменше для двох інвокацій;
- cleanup-флоу задокументований і реально видаляє створені ресурси.

### Рівень 4 (бонус, після Рівня 3)
Додати автотести якості відповіді агента:
- інтеграція `DeepEval`;
- інтеграція `RAGAS`;
- збереження результатів автотестів і короткий висновок по метриках.

## Обов'язковий обсяг
Для проходження домашнього завдання обов'язково виконати Рівні 1-3.

## Definition Of Done
### Pass criteria (mandatory)
1. Повністю виконано Рівень 1 (test cases написані і покривають ключові сценарії).
2. Повністю виконано Рівень 2 (є зрозумілий test guide та стратегія роботи з датасетами).
3. Повністю виконано Рівень 3 (є робочий деплой, докази запуску тестів, observability evidence і cleanup).
4. End-to-end флоу відтворюється з чистого середовища за документацією.
5. Перший invoke повертає consent challenge або коректно обробляється вже наданий consent.
6. Другий invoke повертає валідну відповідь із `sources` і trace tool-виклику.

### Bonus criteria (optional)
1. Рівень 4 виконано: автотести DeepEval/RAGAS додані і результати задокументовані.
2. Записано відео-демо виконання тестів англійською мовою (5-7 хв).

## Що здати
### Обов'язково (для pass)
1. Репозиторій з кодом.
2. Набір test cases (Рівень 1).
3. `README.md` або окремий тестовий гайд (Рівень 2), що містить:
   - runbook для запуску тестування;
   - підхід до різних датасетів;
   - очікувані результати та критерії валідації.
4. `EVIDENCE.md` або еквівалентний документ з:
   - виводом першого invoke;
   - виводом другого invoke;
   - observability-доказами;
   - підтвердженням cleanup.

### Додатково (bonus)
1. Артефакти Рівня 4:
   - конфігурація й запуск автотестів `DeepEval` і `RAGAS`;
   - звіт/вивід результатів метрик;
   - коротка інтерпретація результатів.
2. Відео-демо англійською (5-7 хв) з реальним запуском тестів.

## Рубрика оцінювання
### Базова оцінка (100, mandatory)
1. Якість і повнота test cases (Рівень 1): 50
2. Якість тестової документації та роботи з датасетами (Рівень 2): 25
3. Коректність деплою, E2E-виконання та cleanup (Рівень 3): 20
4. Якість observability evidence і відтворюваність інструкцій: 5

### Бонус (+20, optional)
1. Рівень 4: автотести з `DeepEval` і `RAGAS`: +10
2. Відео-демо англійською з реальним запуском тестів: +10

## Чекліст для ментора

### Mandatory: Рівні 1-3

### 1) Відтворюваність
- `README.md` повний та реально виконується.
- Є `.env.example` без реальних секретів.
- Свіжий запуск не потребує ручних недокументованих фіксів.

### 2) Рівень 1: Test cases
- Є чіткий перелік test cases для ключових сценаріїв.
- Покрито як мінімум один negative auth test.
- Покрито runtime/tool-calling сценарій без cloud mock у основному флоу.
- Покрито consent/path або коректну обробку вже виданого consent.
- Є контрольований fallback/error сценарій.

### 3) Рівень 2: Документація тестування і датасетів
- Документація описує покрокове тестування.
- Описано щонайменше декілька типів датасетів або наборів запитів.
- Для кожного датасету вказано очікувану поведінку/результат.
- Є секція troubleshooting для типових помилок.

### 4) Рівень 3: Деплой і тестування в середовищі
- Використовується `BedrockAgentCoreApp` + `@app.entrypoint` + `app.run()`.
- Runtime/Gateway/target у працездатному стані.
- Перший invoke повертає consent challenge або коректно обробляє вже наданий consent.
- Другий invoke повертає валідну відповідь із source links і trace tool-виклику.
- Є докази реального виконання тестів після деплою.
- Є observability-докази щонайменше для двох запусків.
- Cleanup-процедура реально видаляє створені ресурси.

### Bonus: Рівень 4

### 5) Автотести DeepEval + RAGAS
- Додані й запускаються автотести з `DeepEval`.
- Додані й запускаються автотести з `RAGAS`.
- Збережені результати метрик та короткий аналіз.
- Пояснено, як ці тести інтегруються у загальний процес валідації.

### Загальні вимоги для всіх рівнів
- Рішення лишається в межах матеріалів воркшопу.
- Не змінюється базова архітектура (Runtime/Auth/Gateway/OAuth/Observability).
- У репозиторії/історії немає секретів.
- Для mandatory scope не допускається суто локальний або повністю mock-based флоу замість реального deployed E2E.

## Типові Red Flags (швидкі fail-індикатори)

### Критичні red flags для mandatory (Рівні 1-3)
1. Відсутні test cases або вони не покривають ключові сценарії.
2. Немає negative auth test.
3. Документація тестування неповна або не відтворюється з чистого середовища.
4. Немає стратегії роботи з різними датасетами.
5. Немає доказів реального deployed E2E, observability або cleanup.

### Red flags для bonus claim
1. Заявлено Рівень 4, але відсутні реальні запуски `DeepEval`/`RAGAS` і результати метрик.
2. Заявлено bonus video-demo, але надано лише декларативний опис без реального запуску тестів.
